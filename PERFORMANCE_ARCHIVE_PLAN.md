# 速度改善・アーカイブ管理 設計メモ

## 目的

請求確定チェックアプリの保存・照合処理をさらに高速化しつつ、毎月増えるデータを長期運用できる構造にする。

現状は改善後でも保存・照合に体感10秒弱かかる。短期的には5秒前後、設計変更込みでは2〜3秒台を目標にする。

## 現状

- GAS + スプレッドシート構成。
- 保存先は `BILLING_SPREADSHEET_ID` で管理。
- ほのぼのCSV、e集ちゃんCSV、照合結果、操作履歴は毎月増え続ける。
- データが増えるほど、シート全体読み込みや全体書き込みが重くなる。
- 最新ローカル修正では `APP.VERSION` / `UI_BUILD` を `0.3.5` に更新済み。

## すでに入れた主な改善

- e集ちゃん取込・照合後の重複再取得を削減。
- `SpreadsheetApp.openById` の同一実行内キャッシュを追加。
- `importEshuCsv` / `runReconcile` が照合行と比較サマリーを返すようにして、UI側の追加取得を削減。
- 請求整理保存後、再度 `getHonobonoBillingList` を呼ばず、保存結果で画面更新するように変更。
- 請求調整の一括保存で、シート全体の再書き込みではなく変更行中心の書き込みへ寄せた。
- 共通の請求計算コンテキストを追加し、マスタ・ほのぼの・調整・現金マスタの読み直しを減らした。

## まだ重い可能性が高い箇所

### 1. シート全体読み込み

以下の処理は、対象月だけ必要でもシート全体を読む可能性がある。

- `readSheetObjects_`
- `getLatestImportBatch_`
- `getHonobonoImportRows_`
- `getEshuImportRows_`
- `getReconcileResults`
- `countMismatchResults_`
- `getHistory`

特に増え続けるシート:

- `ほのぼの取込`
- `e集ちゃん取込`
- `照合結果`
- `操作履歴`

### 2. マスタ読込

`loadMasterUsers_` は基本情報マスタ・利用終了者マスタを読むため重い可能性がある。

毎回必要なのは主に以下だけ。

- 照合用ID
- 氏名
- フリガナ
- 利用中 / 終了者区分

軽量マスタキャッシュをアプリ側に持つと効果が大きい。

### 3. 照合結果の保存

`runReconcileInternal_` は対象月の照合結果を作り直す。件数が多い月ではまだ重い。

将来的には、保存で変更されたIDだけ照合結果を更新する差分照合が候補。

### 4. 集計・一覧の再計算

請求一覧は、ほのぼの・調整・現金・マスタから毎回再構築している。

対象月ごとの計算済みキャッシュを持つと、画面表示と照合が速くなる。

## 速度改善の優先順位

### Phase 1: 低リスク改善

目的: 現在の構造を大きく変えず、重複処理をさらに削る。

- 処理時間ログを追加する。
- `formatTextColumns_` が広範囲に走らないよう、書いた行だけに限定する。
- `getHistory` は直近100件だけ読める構造にする。
- `getNameConfirmedList` のような同じMapを複数回読む箇所を1回読みにする。
- 照合結果取得は対象月だけに近い読み方へ寄せる。

期待効果: 10秒弱から7秒前後。

### Phase 2: 月別キャッシュ

目的: 画面表示と照合のたびに請求計算を作り直さない。

新規シート案:

```text
請求計算キャッシュ
```

想定ヘッダー:

```text
対象月
更新日時
照合用ID
顧客番号
氏名
フリガナ
請求状態
ほのぼの請求額
追加請求額
最終請求額
入力対象
当月停止
照合対象
警告
```

更新タイミング:

- ほのぼのCSV取込後
- 請求整理保存後
- 現金支払いマスタ変更後
- 氏名確認済み変更後
- 必要に応じて手動再計算

利用タイミング:

- 請求整理画面表示
- e集ちゃん入力対象一覧
- 照合処理
- ダッシュボード集計

期待効果: 5秒前後。

### Phase 3: 差分照合

目的: 請求整理で変更したIDだけ照合結果を更新する。

考え方:

- 保存処理で変更ID一覧を持つ。
- そのIDだけ請求計算キャッシュを更新。
- そのIDだけ照合結果を更新。
- 件数・金額サマリーだけ再集計する。

注意:

- e集ちゃんCSV取込直後は全件照合でよい。
- 請求整理保存時だけ差分照合にする。

期待効果: 変更件数が少ない場合は2〜4秒台。

### Phase 4: 軽量マスタキャッシュ

目的: 外部マスタシートを毎回読まない。

新規シート案:

```text
利用者マスタキャッシュ
```

想定ヘッダー:

```text
照合用ID
顧客番号
氏名
フリガナ
マスタ区分
同期日時
```

同期タイミング:

- 手動同期ボタン
- 1日1回の時間トリガー
- CSV取込前に古ければ同期

注意:

- 基本情報マスタが正で、キャッシュは読み取り高速化用と位置付ける。
- 同期失敗時は外部マスタ直読みにフォールバックする。

## 年次データ管理方針

毎月データが増えるため、運用中スプレッドシートに全年度分を持ち続けない。

### 年次切替案

毎年1月に新しい保存先スプレッドシートを作成し、以後の保存先を切り替える。

流れ:

```text
1月
↓
新しい「請求確定チェック_2027」スプレッドシートを作成
↓
必要なシートと見出しを作成
↓
設定・現金支払いマスタ・必要な氏名確認済みをコピー
↓
BILLING_SPREADSHEET_ID を新しいIDへ切替
↓
以後の取込・保存・照合は新しい保存先へ
```

推奨運用:

- 新年保存先の作成は自動。
- 保存先切替は最初は確認ボタン付きの半自動。
- 前年12月分が未確定なら自動切替しない。

### 年度アーカイブ案

前年データを年度別アーカイブに退避する。

対象:

- `請求月管理`
- `ほのぼの取込`
- `e集ちゃん取込`
- `照合結果`
- `操作履歴`

流れ:

```text
対象年度を選択
↓
未確定月がないか確認
↓
アーカイブ先スプレッドシートへコピー
↓
コピー件数を検証
↓
運用中シートから対象年度行を削除
↓
実行履歴を記録
```

削除前に必ず件数一致を確認する。

## アーカイブ管理アプリ構想

複数アプリで共通のアーカイブ仕様を使うなら、将来的にまとめアプリを作る価値が高い。

### 役割分担

各業務アプリ:

- 日常の入力・取込・照合・保存を担当。
- 共通仕様に沿ったアーカイブ関数を持つ。

アーカイブ管理アプリ:

- 各アプリの保存先・アーカイブ先・状態を一覧管理。
- 年次保存先作成を実行。
- 年度アーカイブを実行。
- 実行ログと失敗状況を管理。

### 管理アプリの設定シート案

```text
アプリID
アプリ名
スクリプトID
現在の保存先ID
次年度保存先ID
アーカイブ方式
対象シート
年度開始月
最終チェック日時
最終実行日時
状態
メモ
```

### 管理アプリから呼べる共通関数案

各業務アプリに以下の関数を用意する。

```js
getArchiveStatus()
prepareNextYearStorage(year)
validateArchiveTarget(fiscalYear)
runArchive(fiscalYear, options)
switchBillingStorage(spreadsheetId)
getArchiveHistory()
```

返却形式はJSONに統一する。

例:

```js
{
  success: true,
  appName: '請求確定チェック',
  currentStorageId: '...',
  nextStorageId: '...',
  targetYear: 2027,
  uncheckedMonths: [],
  archiveReady: true,
  message: ''
}
```

### 段階的な導入順

1. 各アプリに共通アーカイブ設定・共通関数名を入れる。
2. 各アプリ内で手動アーカイブを実行できるようにする。
3. 管理アプリから各アプリの状態だけ見られるようにする。
4. 管理アプリから新年保存先作成を実行できるようにする。
5. 管理アプリから年度アーカイブを実行できるようにする。
6. 問題なければ時間トリガーで事前チェックや自動作成を行う。

## 注意点

- 請求データなので、アーカイブ後の削除はコピー件数検証後に行う。
- `BILLING_SPREADSHEET_ID` の切替前に未確定月を確認する。
- アーカイブ先URLやIDは必ず履歴に残す。
- 管理アプリから実行する場合も、各業務アプリ側に安全チェックを残す。
- 最初から完全自動削除にしない。確認付き運用から始める。
